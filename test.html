Context

課題と背景

社内の文書はすべて DocuWorks の独自形式（.xdw）で蓄積されていた。専用ソフトがなければ開けず、公式サポートは終了、顧客への納品時には毎回PDFへの変換が発生する。

AIがなかったわけではない。Gemini はすでにあり、現場では自発的にAIを使った効率化が始まっていた。それでも書類は独自形式に閉じたままで、AIに渡せるファイルが一枚もなかった。使う側は634名、ITリテラシーのばらつきも大きい。

02
Tool 01 / 変換

形式を解放する

DocuWorks の独自形式（.xdw）をドロップするだけでPDF化。フォルダごと入れればサブフォルダも自動検出し、処理はPC内で完結する。

Tool 02 / 管理

PDFで管理する

複数PDFをまたぐ階層グループ化、日本式の電子押印、D&D結合まで。書類管理をPDF上で完結させる pdf-annotator。

Tool 03 / 解析

解析にかける

書式がバラバラな保険証券を項目構造（JSON）で抽出し、Excelへ自動転記。人が担っていた転記作業そのものを消す。

JUDGMENT 判断
Point01

足りないのはAIではなく、AIが読める材料だった

導入すべきはAIではない。すでにある。作るべきは材料の方だと判断した。サポート終了済みの独自形式に全書類が閉じている状態を放置できないという判断から、誰の指示でもなく着手している。

Point02

Claudeは、話題性ではなく用途で見極めた

会社に費用を出させる前に自分の負担で検証し、業務資料を作って代表に提示。結論は「Geminiの置き換えではなく、役割が違う」。設計はFigma、実装はClaude、日常の分析はGeminiに分担した。

Point03

設計の順番と、着手の順番を分けた

到達点と4工程は着手前から見えていた。だが最初に作ったのは02。変換自体は非効率の問題に過ぎず、PDF後の編集・階層管理・押印はそもそも手段が存在しなかった。不可能な方から潰した。

Point04

プレゼンで、スコープが変わった

02を作り代表にプレゼンした結果、一括変換ツールの開発を依頼された。個人の危機感として始まったものが、そこで組織の要件になっている。01はこうして生まれた。

Point05

新しい操作を、一切発明しないと決めた

最大の障害は機能不足ではなく「新しい操作を覚えること」だった。ショートカットもUndo/RedoもAdobe・Officeと同じ手の動き。dpiは用途の言葉に翻訳。ファイルはドロップするだけにした。

PROCESS 工程
Step01

形式を解放する

DocuWorks → PDF 一括変換ツール
Python / Flask / xdwlib / PyInstaller

ドロップするだけ。フォルダごと入れればサブフォルダも自動検出。処理はPC内で完結し、データは一切外に出ない。専用ソフトなしで開ける状態になる。

Step02

PDFで管理する

pdf-annotator
pdf.js / fabric.js / jsPDF（フレームワーク不使用）

複数PDFをまたぐ階層グループ化をAcrobat互換のしおりへ書き出し。日本式の電子押印、D&D結合。01からワンクリックで接続し、書類管理がPDF上で完結する。

Step03

解析にかける

保険証券解析システム
React / Claude API / SheetJS

PDFで整った書類を投入。保険会社ごとに書式がバラバラでも、項目構造（JSON）で抽出してExcelへ自動転記する。手作業の転記が消える。

Step04
